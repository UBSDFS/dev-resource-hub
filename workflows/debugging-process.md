# Debugging Process

## Purpose

Provide a structured approach for identifying, isolating, and resolving software issues.

A systematic debugging process reduces wasted time and encourages problem-solving through observation, analysis, and testing.

---

## 1. Understand the Problem

- What is happening?
- What should be happening?
- Can the issue be reproduced consistently?

---

## 2. Gather Information

- Read the error message.
- Review logs.
- Check browser console.
- Review terminal output.
- Identify recent code changes.

---

## 3. Isolate the Problem

- Narrow the scope.
- Identify the file, function, or component involved.
- Verify inputs and outputs.
- Reproduce the issue using the smallest possible example.

---

## 4. Form a Hypothesis

Ask:

"What is the most likely cause?"

Avoid making multiple changes at once.

Change one thing.

Test.

Observe.

Repeat.

---

## 5. Implement the Fix

- Make one logical change.
- Keep changes small.
- Verify the fix addresses the root cause.

---

## 6. Test Again

- Confirm the original issue is resolved.
- Test related functionality.
- Look for unintended side effects.

---

## 7. Document

If the issue revealed a new lesson, workflow, or best practice, add it to the Dev Resource Hub.

---

## Common Questions

Before searching online, ask:

- Did I read the error message?
- What changed most recently?
- Can I reproduce the issue?
- Is this a syntax, logic, configuration, or environment problem?
- Have I verified my assumptions?

---

## Guiding Principle

Debugging is a process of investigation, not guesswork.

Understand the problem.

Change one thing at a time.

Observe the results.

Repeat until the root cause is identified.
